徹頭徹尾的重構，
不是換語言的轉譯。

我們的 agent harness，與兩套個人型開源參考（OpenClaw、Hermes）逐項技術對照。 三套都是 agent harness，好的機制我們有借鏡；但它們是個人單機助手，我們是多租戶 SaaS—— 身份、資料、計量、並發模型全部為「多租戶 × 企業 × serverless」從地基重新設計。

對照對象 · OpenClaw（TypeScript / Node，CLI + daemon）　Hermes（Python，CLI + gateway）　Pandora（TypeScript / Next.js on GCP serverless）
01

一句話定調：機制可借鏡，形態必重寫

它們 · A

個人單機助手

一個常駐程序服務一個主人，跑在你自己的裝置上。身份靠「這台機器就是你」，狀態放全域沒問題。

我們 · B

多租戶 SaaS

無狀態 serverless，一個服務同時切割服務很多帳號。身份必須隨每次呼叫帶、資料必須逐租戶隔離。

關鍵反證

同語言，仍不照抄

OpenClaw 本來就是 TypeScript / Node，跟我們同語言。若只是「轉譯」，照抄最省事——但它的單程序模型放到 serverless 會跨租戶污染，所以整個重寫。

03

三個決定性差異（放大看）

表格裡最關鍵的三列，攤開來看「為什麼不能照抄」。

差異 A · 執行模型

常駐有狀態程序 ↔ 無狀態 serverless 函數

OpenClaw / Hermes

一個常駐程序從開機活到關機，狀態（session、記憶、credential）就放在程序記憶體與本機檔案裡。一個程序＝一個人。

Pandora

請求進來才活、回完就可能被回收；同一容器同時跑 80 個併發請求、橫向擴到多 instance。沒有「常駐記憶體」可以放身份。

apphosting.yaml · maxInstances=10 concurrency=80 · 無狀態容器
差異 B · 身份模型

沒有 userId ↔ UID 是分割鍵

OpenClaw / Hermes

OpenClaw 的訊息物件根本沒有 userId 欄位；Hermes 的 user_id 是可有可無的標籤，不參與授權、不是分割鍵。

// openclaw createUserMessage() { role:"user", content, timestamp } // ← 無 userId
Pandora

身份隨每一次工具呼叫夾帶；工具內部一律讀 context.userId，缺身份就 loud-fail，絕不偷用全域舊值。

execute(args, context) // 讀 context.userId，per-UID scope // 無 context → throw
差異 C · 並發安全（為什麼照抄會出事）

單人無併發假設 ↔ per-call context 隔離

OpenClaw / Hermes

因為假設「只有你一個人」，把身份放全域單例完全沒問題——這在單機是乾淨的設計，不是缺點。

Pandora

同一容器 80 併發下，全域身份會被後一個請求蓋掉前一個，造成跨租戶污染。所以我們刪掉全域、改用 per-call context，並加租戶隔離回歸測試守住。

smoke:tenant-isolation // 驗證無 fallback 全域路徑被走到

05

那我們向參考學了什麼？

學的是「好點子」，不是「整套搬過來」

誠實說：好的機制我們有借鏡——但都重新實作、並改造成多租戶形態。學機制、不學形態。
  • 工具契約 + 動態註冊的形態（一個 tool 抽象 + registry，三家共通）
  • ReAct 執行迴圈 + 並行工具呼叫的骨架
  • MCP 工具命名與 client 形態（啟發 source / exposure 切分）
  • 記憶「先召回、後寫入」的回合紀律（pre-turn / post-turn）
同樣是 agent harness，
但這是不同物種，不是同物種換語言。

為 GCP serverless 多租戶從地基重寫——身份是分割鍵、狀態靠 per-call context、 計量 / 配額 / 權限是第一公民。參考專案給了好的起點與機制； 但要能當成企業級 SaaS 被很多帳號同時、安全地使用，是另一套從頭打造的工程。
