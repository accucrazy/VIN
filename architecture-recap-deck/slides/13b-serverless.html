架構與成本 · ARCHITECTURE & COST / Serverless

不靠「一人一台 VM」，改靠 serverless 共用

個人型 agent 用「物理隔離」——一人一程序／VM／sandbox，單人很乾淨；但要當 SaaS 服務大量用戶，這條路在成本與擴縮上撐不住。

個人型的隔離方式
靠「物理隔離」

一人一 VM / sandbox

→ 每個用戶各開一份環境
→ 用戶數一多，成本 ×N
→ 冷啟動慢、資源閒置
→ 規模化：撐不住
→
我們 · GCP Serverless 多租戶共用
單一服務、多租戶共用，隔離靠軟體層
1
每次呼叫各自帶身份
取代全域單例，多帳號併發不互相污染。
2
per-UID 計量與配額
每個帳號的用量、月上限各自獨立。
3
per-tenant 配置
技能 / 工具 / 記憶 / 資料存取權限，逐租戶各自 scope。
為什麼重要 隔離成本攤在程式邏輯裡，而不是乘上 VM 數量——這正是能把 harness 做成SaaS 服務的關鍵。
技術細節 · 建在 GCP serverless（App Hosting / Cloud Run，無狀態、按需擴縮、按量計費）；正因「單程序、多 UID 併發」，更不能沿用個人型的全域狀態。per-tenant 配置那層即第 16 頁 Control Plane 的 TenantScope。
